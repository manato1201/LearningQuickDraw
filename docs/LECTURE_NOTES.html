第1講

描画の基本単位 — ピクセルとは何か

この講義で私が伝えたいのは「なぜそうなっているか」です。GPUが何をやっているかを教科書で読むより、CPUで1ピクセルずつ手で書く方が100倍速く理解できます。

「GPUがやっていることの本質を理解したければ、まずCPUで同じことを自分で実装しろ」— 実務での鉄則

1.1 フレームバッファ

画面に何かを表示するということは、メモリ上の配列に数値を書くことです。

// 640x480 ピクセルのフレームバッファ
uint32_t pixels[640 * 480];   // = 1,228,800 バイト ≒ 1.2MB

// ピクセル (x, y) に色を書く
pixels[y * 640 + x] = 0xFFRRGGBB;  // ARGB8888 形式

これは QuickDraw の GrafPort portBits（BitMap）と全く同じ構造です。After Effects の PF_EffectWorld も、OpenGL のフレームバッファも、本質は同じ「配列にピクセルを書く」だけです。

640×480、ARGB8888 のフレームバッファを 60fps で書き換えるとき、毎秒何 MB のデータを処理するか？

640 × 480 × 4 bytes = 1,228,800 bytes × 60fps = 73.7 MB/s
1ピクセルあたり: 16.7ms ÷ 307,200px = 54 ナノ秒（約 160 CPU サイクル）

1.2 なぜ最適化が必要か

60fps = 1フレームに 16.7ms しかない

単純な全ピクセル更新: 307,200 回の処理が必要
流体シミュレーション: 1ボクセルあたり数十の演算
→ 最適化なしでは到底リアルタイムに間に合わない
第2講

直線を描く — DDA アルゴリズム

2.1 問題の定式化

「点 (x0,y0) から (x1,y1) までの直線をピクセルで描け」

素朴な解法（なぜダメか）

// NG: ステップ幅 0.001 は恣意的、遅い、穴が開く可能性あり
for (float t = 0; t <= 1.0f; t += 0.001f) {
    setPixel(x0 + (x1-x0)*t, y0 + (y1-y0)*t, color);
}

2.2 DDA の考え方

DDA (Digital Differential Analyzer): 「長い方の軸を主軸にして1ずつ進む」

水平方向が長い (|dx| >= |dy|):
  x を 1 ずつ増やす  → |dx| 回で必ず終わる
  y = y0 + slope * (x - x0)  → 毎ステップ slope を加算するだけ

slope = (y1 - y0) / (x1 - x0)  ← 除算は初期化時に1回だけ！
// QuickDraw 0501DDALine.p の直訳 (C++)
float slope = (float)(y1 - y0) / (float)(x1 - x0);
float fy    = (float)y0;
for (int x = x0; x <= x1; x++) {
    setPixel(x, (int)std::round(fy), color);
    fy += slope;   // 毎ステップ加算のみ
}

2.3 固定小数点最適化

ボトルネック: float の加算と round()。QuickDraw の解決策 — 整数に変換する。

16.16 固定小数点:
  32bit整数の上位16bit = 整数部
  32bit整数の下位16bit = 小数部

  3.5 → 0x00038000  (3*65536 + 32768)

  float加算  → 整数加算 (高速)
  round()    → >> 16   (ビットシフト1回、超高速)
// QuickDraw 0503DDAFixedLineSpeed.p の直訳
const int SHIFT = 16;
int slope = (dv << SHIFT) / dh;   // 除算1回のみ
int fy    = y0 << SHIFT;
for (int x = x0; x <= x1; x++) {
    setPixel(x, fy >> SHIFT, color);   // >> 16 = round() の代替
    fy += slope;                        // 整数加算のみ (最速)
}
QuickDraw 0519 Arith64.asm は Motorola 68000 (32bit CPU) で 64bit 演算を ADDX.L（キャリー付き加算）命令で自前実装していた。 「ハードウェアの限界を知り、それを回避する」実務的思考の原点。

点 (0,0) → (6,4) を DDA で描くとき、各 x での y 座標を求めよ。
（slope = 4/6 ≈ 0.667）

x=0: 0.0→0, x=1: 0.667→1, x=2: 1.333→1, x=3: 2.0→2, x=4: 2.667→3, x=5: 3.333→3, x=6: 4.0→4

第3講

三角形を塗る — スラブラスタライズ

3.1 なぜ三角形か

現代の GPU は全てのジオメトリを三角形に分解して描画します。理由: ①3点は必ず平面上にある ②凸図形なので塗りつぶしが単純 ③並列処理に最適。

3.2 スラブ（Slab）描画の原理

      * top(x0,y0)
     / \
    /---\  ← スキャンライン（スラブ）1本
   /-----\
  *-------+  ← mid(x1,y1)
  |       |
  *-------*  ← bottom(x2,y2)

アルゴリズム:
  1. 頂点を Y 順にソート (top/mid/bottom)
  2. 長辺 (top→bottom) と短辺 (top→mid, mid→bottom) を特定
  3. y = top から bottom まで 1 ずつ:
       xa = 長辺上の X
       xb = 短辺上の X
       for x = xa to xb: setPixel(x, y)  ← スラブ1本
GPU のラスタライザも本質的にスキャンライン変換（スラブ）を行う。 Phase 2 では同じキューブを描くとき GPU がこれを秒間 10億回以上並列実行する。

3.3 Z バッファ（深度テスト）

// 深度バッファ: ピクセルと同解像度の float 配列
float depth[W * H];  // 初期値: FLT_MAX（無限遠）

bool testAndSetDepth(int x, int y, float z) {
    if (z < depth[y*W + x]) {
        depth[y*W + x] = z;
        return true;   // 手前 → 描画許可
    }
    return false;      // 奥   → スキップ
}

スキャンライン内での Z 線形補間:

float frac = (float)(x - xa) / (float)(xb - xa);
float z    = za + (zb - za) * frac;   // 左端Z と右端Z の間を補間
第4講

3D 空間 — 変換行列と透視投影

4.1 回転行列 (Pitch / Yaw / Roll)

X軸回転(Pitch):       Y軸回転(Yaw):        Z軸回転(Roll):
| 1   0    0  |       | c  0  s |           | c -s  0 |
| 0  cos -sin |       | 0  1  0 |           | s  c  0 |
| 0  sin  cos |       |-s  0  c |           | 0  0  1 |

合成: rot = rotX(pitch) * rotY(yaw) * rotZ(roll)
適用: v_out = rot * v_in
行列の積は順序依存（非可換）。rotX * rotY ≠ rotY * rotX
これが「ジンバルロック」の根源。実務では Quaternion を使うことが多い。

4.2 透視投影

焦点距離: f = 1 / tan(fov / 2)

投影:
  px = (v.x * f) / z   ← 遠いほど小さく見える
  py = (v.y * f) / z

スクリーン変換:
  sx = px * W/2 + W/2
  sy = -py * H/2 + H/2  (Y 軸反転: スクリーンは下方向がプラス)
fov=60°, z=4.0 のとき f はいくつか？
f = 1/tan(30°) = 1/0.577 ≈ 1.732

クリッピングと領域最適化

5.1 クリッピングの本質

「見えない部分は描かない」— これが最大の最適化。

// QuickDraw Ch3: SetClip の実装
void putPixel(int x, int y) {
    if (x < clip_x || x >= clip_x + clip_w) return;  // 範囲外 → スキップ
    if (y < clip_y || y >= clip_y + clip_h) return;
    pixels[y * W + x] = color;
}

5.2 ダーティレクト（Dirty Rect）

「変化した領域だけ再描画する」— 動画・ゲーム・UIで最も効果的な最適化。

フレームn:    □□□□□
フレームn+1:  □□■□□   ← 1箇所だけ変化（5%）

ダーティレクト なし: 100% 再描画
ダーティレクト あり:   5% 再描画  →  95% 削減
# numpy ベクトル演算で高速差分計算
diff          = np.abs(current - previous) > threshold
changed_count = diff.sum()
skip_ratio    = 1.0 - changed_count / diff.size
# Phase 3-3 実測: 96.9% スキップ達成

5.3 BVH（階層バウンディングボリューム）

BVH ツリー:
  ルート（全体 AABB）
    ├─ 左子（左半分）
    │    ├─ 葉（アイテム 1-4）
    │    └─ 葉（アイテム 5-8）
    └─ 右子（右半分）  ← 視野外なら このサブツリー全体をスキップ

視錐台カリング: O(log N) で可視アイテムを抽出
Phase 3-3 実測: 125アイテム / Depth 7 / 0.18ms で構築

第7講

最適化の実践 — LOD と LRU

7.1 LOD（Level of Detail）

距離  2 → LOD0 フル解像度  (64³)   処理量: 1
距離 15 → LOD1 1/2 解像度  (32³)   処理量: 1/8    ← 12.5%
距離 40 → LOD2 1/4 解像度  (16³)   処理量: 1/64   ← 1.6%
距離 80 → LOD3 1/8 解像度  (8³)    処理量: 1/512  ← 0.2%

1辺が 1/2 → x方向 1/2 × y方向 1/2 × z方向 1/2 = 1/8
# numpy ストライドスライスで O(1) に近い速度
def downsample(data, lod):
    step = 2 ** lod
    return data[::step, ::step, ::step]  # コピーなし、参照のみ

7.2 LRU キャッシュ

from collections import OrderedDict

class LRUCache:
    def __init__(self, max_size=10):
        self._cache = OrderedDict()
        self._max   = max_size

    def get(self, key):
        if key in self._cache:
            self._cache.move_to_end(key)   # 最近使用済みに更新
            return self._cache[key]
        return None

    def put(self, key, value):
        if len(self._cache) >= self._max:
            self._cache.popitem(last=False)  # 最も古いものを破棄
        self._cache[key] = value

第9講

アルゴリズムの選び方 — 実務の視点

「推測で最適化するな。計測してから最適化せよ」— Donald Knuth
QuickDraw 0502 はまず計測コード（TickCount）を書いてから最適化している。

最適化の優先順位

  • 最大効果描かない（フラスタムカリング・クリッピング）
  • 大効果まとめて描かない（LOD・BVH）
  • 中効果変化した部分だけ描く（ダーティレクト）
  • 中効果計算を安く済ませる（固定小数点・SIMD）
  • 小効果メモリ効率を上げる（LRU キャッシュ）
  • 最後に並列化（マルチスレッド・GPU）

第10講

まとめと次のステップ

この講義で学んだこと

  • 描画の基本は配列にピクセルを書くだけ
  • DDA: 除算1回→加算のみで直線を描く / 固定小数点でさらに高速化
  • スラブ: 水平帯を積み重ねて三角形を塗る（GPUラスタライザと同じ）
  • Zバッファ: 深度テストで奥行きを正確に処理（スキャンライン上で補間）
  • 変換行列: 3D変換を4×4行列で統一的に表現（Pitch/Yaw/Roll）
  • クリッピング: 見えない部分は最初から計算しない（最大の最適化）
  • ダーティレクト: 変化した部分だけ更新（Phase 3-3 実測 96.9% スキップ）
  • BVH: 空間を階層化して O(log N) でクエリ
  • GPU: CPU でやっていた処理を数万コアで並列化した結果
  • 実務: まず計測し、効果の大きいところから最適化する

基礎2. 三角形 (10,0), (0,20), (20,20) をスラブ法で塗るとき、y=10 での xa, xb を計算せよ。

応用3. Phase 1 の fillTriangleZ() に Lambert シェーディングを追加せよ。

応用4. Phase 3-3 の BVHTree に球クエリ（球内のアイテムを返す）を実装せよ。

基礎1. 2.75 = 2 × 65536 + 49152 = 0x0002C000（180224）

基礎2. 頂点ソート後: top=(10,0), mid=(0,20), bottom=(20,20)。y=10 は上半分。長辺(10,0)→(20,20): x=10+(20-10)*10/20=15。短辺(10,0)→(0,20): x=10+(0-10)*10/20=5。xa=5, xb=15
